# 存链（tuku）运维与恢复手册

> 出事时照这份做。所有 CF 操作都在**本机 PowerShell**跑（沙箱连不到 CF）；跑 wrangler 前先 `npx wrangler whoami` 确认是 **Hannah** 号。执行目录：`D:\系统默认\桌面\畅密\tuku`。

## 1. 代码回滚（部署了坏版本）
CI 部署走 GitHub Actions。回滚两种方式：
- **重部旧提交**：`git revert <坏提交>` 或 `git checkout <好提交> -- index.js` 后提交，再手动触发 `deploy.yml`（双击 `一键部署.cmd`）。
- **Worker 版本回滚**（最快，不改代码）：`npx wrangler rollback`（回到上一个已部署版本）；或先 `npx wrangler deployments list` 看版本，再 `npx wrangler rollback --message "回滚原因"`。

## 2. 数据库恢复（D1 tuku-db 误删/误改）
D1 自带 **Time Travel**（默认保留约 30 天，可回到任意时间点）：
- 看当前书签：`npx wrangler d1 time-travel info tuku-db`
- 恢复到某时刻：`npx wrangler d1 time-travel restore tuku-db --timestamp="2026-07-12T10:00:00Z"`
  （或用 `--bookmark=<书签>`）。**先在测试意识下确认时间点**，恢复不可逆地覆盖当前。
- 导出备份（建议定期手动做一次存档）：`npx wrangler d1 export tuku-db --remote --output=tuku-db-backup-YYYYMMDD.sql`

## 3. 建库 / 迁移（灾备重建或加表）
- **从零重建**：`npx wrangler d1 execute tuku-db --remote --file=schema.sql`（`schema.sql` 已是含所有列的权威建表脚本）。
- **老库增量升级**（按顺序，各只跑一次；报 duplicate column 忽略即可）：
  - `npx wrangler d1 execute tuku-db --remote --file=migration-v2.sql`（byte_limit / kind / r2_key / mime）
  - `npx wrangler d1 execute tuku-db --remote --file=migration-v3.sql`（pending_deletes 对账表）

## 4. 存储（R2 / CF Images）
- **R2 桶 `tuku-files`**：非图片文件。建议在 CF 后台给桶配一条**生命周期规则**清理未完成的分片上传（AbortIncompleteMultipartUpload，如 7 天），避免断点续传残留占空间。
- **CF Images**：图片本体。删除失败的孤儿由 `pending_deletes` + 每 6 小时的 `scheduled` 对账补删（见 index.js 的 `scheduled` 与 `recordPending`）。
- 想手动看对账队列积压：`npx wrangler d1 execute tuku-db --remote --command "SELECT kind,count(*) FROM pending_deletes GROUP BY kind"`

## 5. 机密（secret）丢失/轮换
`npx wrangler secret put <名字>`（值本机手敲，不进 git）。四个：`CF_IMAGES_TOKEN`、`SESSION_SECRET`、`APP_KEY_TU_BASIC`、`APP_KEY_TU_PRO`。
- 注意：轮换 `SESSION_SECRET` 会让**所有在线客户被登出**（旧会话签名失效），非必要不动。

## 6. 定位问题
- 实时日志：`npx wrangler tail tuku`（或 CF 后台 Workers → tuku → Logs，observability 已开）。
- 健康检查：`GET https://tuku.5209696.xyz/health` 返回版本号即存活。
